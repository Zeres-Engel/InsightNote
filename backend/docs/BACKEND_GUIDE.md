# ⚙️ InsightNote - Backend Architecture & Integration Guide

Welcome to the **InsightNote** Backend architecture guide. This document details the server-side architecture, multi-notebook routing logic, database integrations, and ZeRAG core pipeline.

---

## 🏗️ 1. System Architecture & Database Integrations

InsightNote leverages a **Hybrid Multi-Database** storage strategy to power high-fidelity GraphRAG operations:

```txt
                       ┌─────────────────────────┐
                       │   FastAPI Web Server    │
                       └───────────┬─────────────┘
          ┌────────────────────────┼────────────────────────┐
          ▼                        ▼                        ▼
┌──────────────────┐     ┌──────────────────┐     ┌──────────────────┐
│     MongoDB      │     │  Neo4j (DozerDB) │     │      Qdrant      │
│  (Doc Metadata   │     │ (Chunk & Entity  │     │  (Dense Vector   │
│  & Status Store) │     │  Relation Graph) │     │   Embeddings)    │
└──────────────────┘     └──────────────────┘     └──────────────────┘
```

1.  **MongoDB**: Acts as the metadata store and document registry. It tracks upload file paths, raw text contents, and handles progressive pipeline status monitoring.
2.  **Neo4j (DozerDB)**: Serves as the Property Graph database. It stores the semantic relationships (e.g., `Customer -[OWNS]-> Policy`) and document chunk connection links.
3.  **Qdrant**: High-performance vector database. Stores 1536-dimensional chunk embeddings (generated by OpenAI or custom embedding bindings) to support dense vector similarity search.

## 🔌 3. Core Routing Mechanics (`insightnote_routes.py`)

All multi-notebook logic is isolated under `/api/notebooks/{notebook_id}`.

### 1. In-Memory Store (`notebooks_db`)
For rapid dev cycles and lightweight demo setups, notebook meta-statuses are stored in a thread-safe module dictionary:
```python
notebooks_db: Dict[str, Dict[str, Any]] = {
    "notebook_insurance_demo": {
        "id": "notebook_insurance_demo",
        "name": "Insurance Analysis (Demo)",
        "source_count": 1,
        "status": "ready"
    }
}
```

### 2. Multi-Notebook API Isolation
*   **List Notebooks** (`GET /api/notebooks`): Returns notebooks in the store.
*   **Create Notebook** (`POST /api/notebooks`): Normalizes requested name into a unique snake-case ID.
*   **Delete Notebook** (`DELETE /api/notebooks/{notebook_id}`): Wipes notebook metadata from the store.

### 3. Progressive Pipeline Tracker
*   When files are uploaded or loaded via `/load-example`, the notebook transitions to `"processing"`.
*   A background thread enqueues the document.
*   The frontend polls `/api/pipeline/jobs/{job_id}` to track processing statuses (`load_file`, `pdf_parse`, `chunking`, etc.) until completion, at which point the notebook status transitions to `"ready"`.

### 4. Delete Source Document (`DELETE /api/notebooks/{notebook_id}/sources/{source_id}`)
Decrements `source_count` in metadata. In a live system, this also runs Cypher deletes (`DETACH DELETE`) on Neo4j for nodes matching the document ID and issues filter deletes on Qdrant points.

## 🧠 4. ZeRAG Core Query Logic (`zerag.py`)

When `/api/notebooks/{notebook_id}/chat` is invoked, the request is directed to `rag.aquery_llm(request.message, param=param)`:

1.  **Keyword Extraction**: Extracts semantic keywords (high-level topics and low-level entities) from the user's prompt.
2.  **Dense Retrieval (Qdrant)**: Executes cosine-similarity searches on chunk embeddings in Qdrant to find the top $k$ relevant context blocks.
3.  **Knowledge Graph Traversal (Neo4j)**: Uses extracted keywords to run graph queries on Neo4j, traversing paths starting from identified entities to explore related rules, clauses, and exclusion parameters.
4.  **Hybrid Reranking (Reranker)**: Combines vector chunks and graph triplets, scores them with a cross-encoder reranker, and filters out low-relevance candidates.
5.  **LLM Generation**: Feeds the optimized context, conversations history, and grounding citations to the LLM (OpenAI/Qwen) to generate a precise, hallucination-free response, returning the reasoning nodes for 3D graph path highlighting.

## ⚡ 5. Execution & Verification Guides

### 1. Active GPU Environment (`gpu_env`)
To perform PDF OCR parsing (MinerU) and vector embeddings generation locally, developers **MUST** activate the GPU environment:
```bash
conda activate gpu_env
```

### 2. Run Backend Individually
To spin up the FastAPI uvicorn developer server locally on port `8000`:
```bash
cd backend
python server.py
```
*   **Interactive Docs**: `http://localhost:8000/docs` (Swagger UI)
*   **Raw JSON Health**: `http://localhost:8000/api/health`

### 3. Run Unit Tests
To verify backend routing compilation and API schemas:
```bash
pytest backend/tests/unit/test_insightnote_routes.py -v
```
Ensure all routes return `200 OK` on standard payloads.
